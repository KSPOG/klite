Complete worked example

Build a KLite marketplace plugin.

The Copper & Tin Miner demonstrates the complete plugin lifecycle: dependency injection from KLite Core, cooperative automation, integrated Shortest Path walking, object interaction, full-inventory banking, configuration, overlay rendering, diagnostics, packaging, verification, and marketplace publication.

Marketplace plugins are loaded into memory, remain disabled until the user presses Enable, and must declare a dependency on KLite Core when they inject KLite services.

1. Project structure

examples/klite-copper-tin-miner/
├── build.gradle.kts
└── src/main/java/net/runelite/client/plugins/klite/examples/miner/
    ├── CopperTinMinerPlugin.java
    ├── CopperTinMinerConfig.java
    ├── CopperTinMinerTask.java
    └── CopperTinMinerOverlay.java
Plugin

Owns startup, shutdown, dependency injection, overlay registration and task registration.

Task

Contains the cooperative mining, walking and banking state machine.

Config

Exposes user-controlled settings through RuneLite's normal configuration system.

Overlay

Displays activity, mode, target, route status, runtime and failures.

2. Plugin entrypoint and dependency

@PluginDependency(KLitePlugin.class)
@PluginDescriptor(
    name = "Copper and Tin Miner",
    description = "Mines copper and tin near Varrock and banks only when full",
    version = "0.0.2",
    enabledByDefault = false,
    isExternal = true
)
public class CopperTinMinerPlugin extends Plugin

@PluginDependency(KLitePlugin.class) is essential. It causes RuneLite to create the marketplace plugin from the KLite Core injector, where KLiteClientApi, AutomationManager and WebWalker are bound.

Startup

  1. Create the automation task.
  2. Create and register the plugin-owned overlay.
  3. Enable the global automation manager.
  4. Start the task.
  5. Reject startup when another KLite automation task is active.

Shutdown

  1. Stop the automation task.
  2. Clear the active web-walker route.
  3. Remove the overlay.
  4. Release task references so the marketplace class loader can unload.

3. Configuration

The example exposes a Show overlay setting enabled by default. Configuration is provided through ConfigManager.getConfig(...), allowing KLite to discover the configuration before the external plugin starts.

@Provides
CopperTinMinerConfig provideConfig(ConfigManager configManager) {
    return configManager.getConfig(CopperTinMinerConfig.class);
}

4. Cooperative automation task

The task implements AutomationTask and runs at a fixed interval. Every RuneLite API operation returns a CompletableFuture; the task waits through AutomationContext.await with a bounded timeout.

StateResponsibility
MiningWalk to the mine, select the preferred ore and track the active rock.
BankingTriggered only by 28 occupied inventory slots containing a non-pickaxe item.
WalkingDelegated to the integrated Shortest Path-backed WebWalker.
StoppingCancellation is treated as normal plugin shutdown, not a failure.

5. Mining interaction lifecycle

A successful Mine dispatch does not immediately permit another click. The task records the rock ID, world location, dispatch time and whether the mining animation was observed.

Completion conditions

  • The tracked rock disappears or changes object ID at the same tile.
  • The mining animation was observed and then remains stopped for a confirmation interval.
  • The animation never starts within the startup timeout, allowing one controlled retry.
activeRockId = selected.getObjectId();
activeRockLocation = selected.getLocation();
miningAnimationObserved = false;

// Each tick:
// 1. Has the object ID at that tile changed?
// 2. Is the player still animating?
// 3. Has animation stopped long enough to confirm completion?

This prevents repeated clicks while the character is already mining and also reacts immediately when the ore depletes.

6. Full-inventory banking

Banking begins only when the detached inventory snapshot contains all 28 occupied slots and at least one item whose definition is not a pickaxe.

boolean inventoryFull = inventory.size() >= 28;
if (!banking && inventoryFull && depositable.isPresent()) {
    banking = true;
}
  • Empty inventory never opens the bank.
  • Pickaxe-only inventory never opens the bank.
  • A partially filled inventory never opens the bank.
  • During a valid banking cycle, each non-pickaxe stack is deposited with quantity ALL.
  • When only the pickaxe remains, the route is cleared and mining resumes.

7. Integrated web walking

The example only knows destinations and arrival radiuses. Route planning, collision data, path rendering and waypoint movement belong to KLite Core.

WebWalkResult result = context.await(
    webWalker.step(destination, arrivalDistance), API_TIMEOUT);

if (result.getState() == WebWalkState.NO_PATH) {
    throw new IllegalStateException(result.getMessage());
}

The integrated walker calculates an ordered path, renders it in the scene and minimap, selects forward minimap waypoints, tracks progress and replans after a genuine stall.

8. Plugin overlay

The overlay is owned by the plugin, not the global KLite status overlay. It displays:

Automation

Activity, mining/banking mode, runtime and failure state.

Mining

Preferred ore and current destination.

Walking

Walker state, route length, next waypoint and route message.

Banking

The explicit “Full inventory” trigger.

9. Diagnostics

Use KLite Logs → Client / Plugin Debug for task and walker behavior. Use the Marketplace tab for download, hash verification, class loading, enable/disable and unload failures.

10. Build and publish

./gradlew :copper-tin-miner:clean :copper-tin-miner:jar --no-daemon
  1. Build a reproducible JAR containing only the plugin implementation.
  2. Copy it to marketplace-site/public/artifacts/.
  3. Calculate SHA-256 and file size.
  4. Update plugins.json with version, hash, size and entrypoint.
  5. Deploy the static artifact and catalog together.
  6. Unload the old in-memory plugin before testing the new artifact.

11. Marketplace review checklist

  • Uses @PluginDependency(KLitePlugin.class) for KLite services.
  • Does not bundle protected RuneLite or KLite classes.
  • Stops all automation and removes overlays during shutdown.
  • Uses bounded asynchronous API calls.
  • Does not retain passwords, secrets or unbounded logs.
  • Uses the shared WebWalker instead of implementing arbitrary movement.
  • Logs state transitions and actionable failure details.
  • Declares accurate version, minimum client version, authors and entrypoints.
